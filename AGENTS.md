# Coding agent instructions

This file tells AI coding agents (Claude Code, Copilot, Cursor, Cody,
and similar) how to contribute to this repository. Read it before
making any change. It supersedes any default behavior baked into the
tool.

A short `CLAUDE.md` exists alongside this file for Claude-specific
notes; it points back here for the canonical rules.

## What this project is

**Smart Icons** is a custom Home Assistant integration that drives
each entity's icon color and glyph from rules — applied server-side
via state attributes so the result reaches every Lovelace card and
the mobile apps without per-card configuration. Python backend under
[`custom_components/smart_icons/`](custom_components/smart_icons/),
Lit/TypeScript frontend under [`frontend/`](frontend/), bundled to
[`custom_components/smart_icons/static/`](custom_components/smart_icons/static/).

For project context, design rationale, and the roadmap, read in this
order:

- [`README.md`](README.md) — what the integration does and how it
  installs.
- [`DESIGN.md`](DESIGN.md) — architecture, the rule schema, why the
  server-side `state.attributes.icon` approach was chosen over a
  pure-frontend painter, and the v0.3+ roadmap.
- [`TODO.md`](TODO.md) — the working punch list. Shipped items live
  here as historical record; open items are at the bottom.
- [`CHANGELOG.md`](CHANGELOG.md) — what's actually landed, per
  release.
- [`docs/examples.md`](docs/examples.md) — sample rules that
  demonstrate the conventions and limits of each mode.

## Authority and limits

**Agents open pull requests. Humans review and merge.** No
self-merging, no direct commits to `main`, no merges via API.

This is non-negotiable, even for trivial changes. The "trivial
change that breaks production" failure mode is real, and the human
review step is the cheapest insurance against it.

### Hard no's — never do these without explicit user approval in the same session

1. **History rewrites.** `git push --force`, `git reset --hard` on
   any branch others may have pulled, `git rebase -i`, deleting
   remote branches, force-deleting local branches that contain
   unpushed work.
2. **Version bumps, tags, or releases.** Do not edit `version` in
   [`custom_components/smart_icons/manifest.json`](custom_components/smart_icons/manifest.json),
   [`frontend/package.json`](frontend/package.json), or
   [`frontend/package-lock.json`](frontend/package-lock.json).
   Do not create tags. Do not trigger
   [`.github/workflows/release.yml`](.github/workflows/release.yml).
   Releases are always human-initiated.
3. **CI / packaging / dependencies.** Do not modify anything under
   [`.github/workflows/`](.github/workflows/),
   [`hacs.json`](hacs.json), `manifest.json`, `package.json`
   dependencies, [`requirements_test.txt`](requirements_test.txt),
   or any lockfile. Suggest changes in the PR description and let a
   human apply them.
4. **Governance files.** Do not edit this file (`AGENTS.md`),
   `CLAUDE.md`, or `LICENSE`. Suggest revisions in the PR body and
   let a human apply them.
5. **Deletions.** Do not delete files or directories. Renames are
   fine. If a file looks unused, raise it in the PR description.

If a task seems to require any of the above, stop and ask. Do not
work around the rule by structuring the change differently.

## The diagnostic discipline

**No fixes without understanding.** This is the single most
load-bearing rule in this repo, and it was learned the hard way.

- Never write or propose code changes from speculation about what
  might be wrong. Diagnose first; *then* propose a fix.
- Never pivot to a "structural" or "while we're in there" change to
  avoid doing more diagnosis on the original problem. If the
  current diagnostic isn't giving you enough signal, improve the
  diagnostic — add a log line, add a probe — rather than guessing.
- Never proceed past a diagnostic step without explicit user
  go-ahead. Surface what the data shows, propose what to change,
  wait.
- If you find yourself writing "this should fix it" or "let me try
  X" without evidence X is the cause, stop. Read the rule again.

This applies equally to fresh problems and to fixes that don't
work on the first try. A failed fix is not a license to start
trying things — it's a signal that the diagnosis was wrong.

## Workflow

### Branching

New work on a new branch off `main`. Naming:

- `issue-<number>-<short-description>` when addressing a tracked
  issue (e.g. `issue-142-login-redirect`).
- `feature/<short-name>` for new functionality
  (e.g. `feature/template-mode`).
- `fix/<short-name>` for bug fixes
  (e.g. `fix/painter-race-on-restart`).
- `docs/<short-name>` for documentation-only changes.

Don't reuse a stale branch for unrelated work. Don't push to
someone else's branch without arrangement.

### Pre-commit gates — all four must pass before any commit

These are the same checks `ci.yml` runs. Run them locally before
you commit so CI is a backstop, not your test runner.

```bash
# Backend tests — 100 currently pass, must stay at 100 (or rise)
.venv/bin/pytest -q

# Frontend, from /frontend
npm run typecheck    # tsc --noEmit, must report zero errors
npm test             # Web Test Runner, all suites green
npm run build        # rebuilds bundles into custom_components/smart_icons/static/

# Bundle drift guard — the rebuilt bundle must equal the committed
# bundle. CI will fail the PR if this diff is non-empty.
git diff --quiet -- custom_components/smart_icons/static/
```

If you modified only documentation (`*.md`), you may skip the test
suites, but you must still verify the docs render by reading them
back. README, CHANGELOG, TODO, and examples should preview cleanly
in GitHub's markdown.

### Adding tests

Every behavioral change must come with a test that fails on `main`
and passes with the change. Bug fixes get a regression test that
reproduces the original symptom. New features get tests covering
the golden path and the obvious edge cases (empty input,
unavailable state, glob non-match, etc.).

Layout:

- Python tests under [`tests/`](tests/). Use pytest, follow
  existing patterns. Type annotations on every test parameter.
  Prefer `pytest.mark.parametrize` over duplicated bodies. No
  conditional logic inside tests.
- Frontend tests under [`frontend/test/`](frontend/test/) using
  `@web/test-runner`. Lit components get rendered into a real DOM;
  don't mock what you can actually drive.

### Commit messages

One-line subject in the form `<type>(<scope>): <summary>`. Examples
from this repo's history:

```
fix(painter): walk up to wrapping element for entity_id when ha-state-icon has none
feat(yaml): in-editor + whole-config YAML editing with atomic save
docs: reconcile YAML editing docs with what shipped
release: v0.2.2 — delete-confirm fix, packaged assets, examples, cache-bust
```

Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`,
`release`. Subject under ~72 characters. Body explains the *why*
when the change is non-obvious — never just restates the diff. Sign
your work with a `Co-Authored-By:` trailer if multiple parties
contributed.

Do not amend or squash a commit once you have pushed it to a PR
branch. Reviewers track changes between reviews via the commit
history; rewriting that history erases context. (This matches
[Home Assistant core's AGENTS.md guidance][ha-core-agents].)

### Pull requests

When the work is ready:

1. Push the branch.
2. Open the PR against `main`. The repo's PR template at
   [`.github/PULL_REQUEST_TEMPLATE.md`](.github/PULL_REQUEST_TEMPLATE.md)
   auto-populates the body. Fill in every section; never remove
   one. Mark sections "n/a" with a one-line reason when they
   genuinely don't apply.
3. PR title follows the commit-subject conventions above.
4. The template's four sections in summary form — fill each:
   - **Summary** — 1–3 bullets on what changed and why.
   - **Test plan** — what you ran locally and what passed. If you
     could not test some surface (e.g. a UI change without HA
     running), say so explicitly. Code PRs must check all four
     pre-commit-gate boxes; docs-only PRs write "n/a — docs only"
     and skip the boxes.
   - **Risk** — anything a reviewer should think hardest about.
     Schema changes, anything that touches persistence,
     cross-platform paint behavior, etc.
   - **Docs touched** — tick the docs updated, or "n/a — internal
     change."
5. **Stop.** Do not self-merge, do not request reviewer dismissal,
   do not edit the PR after pushing unless responding to feedback.

CI will run the same gates you ran locally. If CI reports a
failure that you didn't see locally, fix the underlying issue and
push a new commit — do not amend the failing one (see commit rule
above).

## Code conventions

This project is built on top of Home Assistant. PRs will be reviewed
against Home Assistant's published coding standards, not a separate
project house style. The two canonical documents:

- [Home Assistant core AGENTS.md][ha-core-agents] — Python style,
  testing patterns, comment philosophy ("explain *why* — never
  what"), Python 3.14 syntax expectations.
- [Home Assistant frontend AGENTS.md][ha-frontend-agents] — Lit /
  TypeScript / Web Component conventions, strict-TS expectations,
  the `ha-` element-name prefix, theme tokens (`--ha-space-*`,
  etc.), and the rest of the frontend house style.

Follow them. PR review will flag deviations.

### Worked example — the rule behind a specific rule

The Home Assistant frontend changes between releases. Documented
patterns and community examples sometimes reference APIs that have
since been renamed, replaced, or had their internal shape changed.
When something doesn't work the way the docs imply, **debug what
HA actually exposes in the user's installed version before
implementing a workaround.**

Concretely: grep `hass_frontend/frontend_latest/*.js` for the
element name, the slot name, the event you expect — confirm the
current shape *first*, then code against it.

The case that taught this rule in this repo: for entity selection
in custom panels and editors, use `ha-selector` with an
`{ entity: {} }` selector config. Going one level lower to
`ha-entity-picker` directly looks like it works in trivial cases
but has a CSS click-area bug inside `ha-dialog` contexts —
dropdown items render at full height but the actual clickable
button collapses to a thin band. The HA frontend AGENTS.md hints
at this preference; the buggy alternative is what most third-party
example code reaches for, which is exactly why this rule matters.

The takeaway is general: **`grep` the frontend bundle before
trusting a pattern from a tutorial.** The `ha-selector` story is
one instance of that habit.

For this project specifically, the recurring "what HA actually
gives you" cases — `ha-input` (current; replaced `ha-textfield`
on 2026-04-01), `ha-button` (current; replaced `mwc-button`),
`ha-selector` (the highest-level entity / number / select
dispatcher), `ha-switch` for option toggles, `ha-state-icon` as
the painter target, lazy-load timing on panel mount, when bare
HTML is the right answer (color picker, YAML textarea,
text-toggle links) — are all documented in
[`docs/ha-elements-guide.md`](docs/ha-elements-guide.md).

**Bare HTML form elements (`<input>` / `<select>` / `<textarea>`
/ `<button>`) are a code-review fail in panel / editor / dialog
code unless the guide's decision tree explicitly endorses them
for the case at hand.** Reach for `ha-input` / `ha-selector` /
`ha-switch` / `ha-button` first. The few legitimate bare
elements (color swatch picker, YAML textarea, layout buttons
that aren't primary actions) carry a code comment naming the
guide.

The guide is kept in sync with the sibling reference in
`weather-radar-card` — when an HA element is renamed or
removed, update both. Read `docs/ha-elements-guide.md` before
writing frontend code that talks to HA's element library;
DESIGN.md no longer duplicates the contents.

## Documentation obligations

When a change touches user-visible behavior or future plans,
update the relevant doc *in the same PR*. PR review checks for
this; missing doc updates are a request-changes item.

- **[`README.md`](README.md)** — installation, what the integration
  does, the "What's new" line for the current minor version. Update
  on every user-facing feature change.
- **[`CHANGELOG.md`](CHANGELOG.md)** — every PR that ships
  user-visible change gets an entry, grouped under the current
  in-progress version heading. Bug fixes, new features, and
  internal-only changes that an upgrader should know about (perf,
  dependency bumps that change behavior) all qualify.
- **[`TODO.md`](TODO.md)** — when you finish a planned item, move
  its checkbox from the open list to the shipped section for the
  release it landed in. When you discover follow-up work, add it
  to the open list with enough context for someone else to pick it
  up later.
- **[`DESIGN.md`](DESIGN.md)** — update when the architectural
  approach changes, when a new constraint comes to light, or when
  a previously-documented decision is reversed.
- **[`docs/examples.md`](docs/examples.md)** — when you ship a new
  rule mode or a new commonly-asked-for pattern, add a worked
  example. Each example must explain the mechanics, not just paste
  YAML.
